# Troubleshooting

| Symptom | Diagnostic commands | Likely cause | Resolution | Validation |
| --- | --- | --- | --- | --- |
| Hardware virtualization disabled | `grep -Ec '(vmx\|svm)' /proc/cpuinfo` | VT-x/AMD-V disabled in host BIOS/UEFI, or running inside a VM without nested virtualization | Enable virtualization in host BIOS/UEFI, or enable nested virtualization if this host is itself a VM | Re-run `make prerequisites`; expect `[PASS] Hardware virtualization detected` |
| VirtualBox kernel driver unavailable | `VBoxManage list vms`; on Linux: `lsmod \| grep vbox` | `vboxdrv` kernel module not loaded/built (common after a kernel upgrade without a matching VirtualBox DKMS rebuild) | Reinstall/rebuild VirtualBox kernel modules for your running kernel (`sudo /sbin/vboxconfig` on many distros), then reboot if needed | `VBoxManage list vms` returns without error |
| KVM conflict | `lsmod \| grep kvm`; `VBoxManage list vms` | KVM modules loaded alongside VirtualBox — usually coexists fine on VirtualBox 7.x, but can occasionally cause VM boot failures | If `VBoxManage list vms` fails or VM boot errors mention virtualization, try `sudo modprobe -r kvm_intel kvm` (or `kvm_amd`) — only if you don't need KVM for anything else running right now | `VBoxManage list vms` responds; VM boots |
| Host-only network creation failure | `VBoxManage list hostonlyifs` | VirtualBox couldn't create/attach the host-only adapter (permissions, driver issue) | Check VirtualBox logs (`~/VirtualBox VMs/<vm>/Logs/VBox.log`); try `VBoxManage hostonlyif create` manually to isolate the failure | `VBoxManage list hostonlyifs` shows `vboxnet0` (or similar) at `192.168.56.1` |
| Host-only IP conflict | `scripts/host/check-prerequisites.sh`; `ping -c1 192.168.56.10` | Another local Vagrant/VirtualBox environment (or another device) already uses `192.168.56.10/.11/.12` — a real risk encountered during this module's own development (see root `docs/DEPENDENCIES.md`) | Power off / `vagrant destroy` the conflicting environment, or change `config/cluster.env`'s IPs to an unused range and re-render | `check-prerequisites.sh` reports `[PASS] No host currently answers on ...` |
| VM boot timeout | `vagrant up otel-control-plane --debug 2>&1 \| tail -100` | Host resource contention (too little RAM/CPU for the profile), slow disk I/O, or a genuinely stuck boot | Check `free -h`/`df -h`; try the `minimum` profile; increase `boot_timeout` in the `Vagrantfile` if boot is merely slow, not stuck | `vagrant up` completes without a `Timeout` error |
| Vagrant SSH timeout | `vagrant ssh otel-control-plane --debug` | VM booted but SSH/guest additions not yet ready, or a networking misconfiguration prevents SSH reaching the VM | Retry after a short wait; check `VBoxManage list hostonlyifs` and that the VM's host-only adapter is actually up | `vagrant ssh otel-control-plane -c true` succeeds |
| Guest Additions mismatch | `vagrant ssh <node> -c "cat /opt/VBoxGuestAdditions-*/version"` vs. `VBoxManage --version` | The `bento/ubuntu-24.04` box's bundled Guest Additions version doesn't match the host's installed VirtualBox version | Usually harmless for shared-folder/basic networking use; if shared folders misbehave, install `vagrant-vbguest` plugin to auto-sync Guest Additions | Shared folder `/vagrant` is readable/writable inside the guest |
| Insufficient memory | `free -h`; `scripts/host/check-prerequisites.sh` | Selected `LAB_PROFILE` needs more RAM than currently available | Use `LAB_PROFILE=minimum`, free up host RAM, or use per-node overrides (`.env`) to reduce allocation further | `check-prerequisites.sh` reports available memory meets the profile requirement |
| Insufficient disk | `df -h ~` | VM disks + box cache + container image layers exceed available space | Free disk space; `vagrant box list` and remove unused boxes (`vagrant box remove`); `docker system prune`-equivalent isn't relevant here (containerd, not Docker) but old `.vagrant/` state from destroyed environments can be cleaned | `check-prerequisites.sh` reports the ~40GB guideline is met |
| Ubuntu package repository failure | `vagrant ssh <node> -c "sudo apt-get update"` | Transient network issue, or a proxy misconfiguration, or a genuinely down upstream mirror | Retry; check `examples/proxy.env.example` if behind a proxy; verify NAT egress works at all (`curl` test from the guest) | `apt-get update` succeeds |
| Enterprise proxy failure | `vagrant ssh <node> -c "cat /etc/apt/apt.conf.d/95proxy"` | `HTTP_PROXY`/`HTTPS_PROXY` set incorrectly, or the proxy requires auth not embedded in the URL | Fix the proxy URL/credentials in your local `.env` (never commit it); confirm the proxy is reachable from the guest's NAT path | `apt-get update` succeeds through the proxy |
| containerd not starting | `vagrant ssh <node> -c "sudo systemctl status containerd"` | Bad `/etc/containerd/config.toml` (e.g. the SystemdCgroup/sandbox_image patch in `03-install-containerd.sh` didn't apply cleanly) | Inspect `/etc/containerd/config.toml` for syntax errors; re-run `containerd config default` and re-apply the documented patches from `config/containerd-config.toml.tpl` | `systemctl is-active containerd` returns `active` |
| CRI disabled | `vagrant ssh <node> -c "sudo crictl info"` | containerd's CRI plugin explicitly disabled in config (shouldn't happen with the default-generated config this module uses, but check if hand-edited) | Ensure `disabled_plugins` in `/etc/containerd/config.toml` does not list the CRI plugin | `crictl info` returns runtime status, not a connection error |
| Incorrect cgroup driver | `vagrant ssh <node> -c "sudo systemctl status kubelet"` (look for cgroup driver mismatch errors) | containerd's `SystemdCgroup` patch didn't apply, while kubelet's `KubeletConfiguration.cgroupDriver: systemd` (set in `config/kubeadm-config.yaml.tpl`) expects it to | Confirm `SystemdCgroup = true` is actually present in `/etc/containerd/config.toml`'s runc options block; restart containerd, then kubelet | kubelet starts without a cgroup-driver-mismatch error in `journalctl -u kubelet` |
| kubelet restarting (before init/join) | `vagrant ssh <node> -c "sudo systemctl status kubelet"` | **Expected and normal** — kubelet has no cluster config until `kubeadm init`/`kubeadm join` runs | No action needed if this is before step `05`/`07`; if it persists *after* init/join succeeded, see the next rows | `is_node_joined` (kubelet.conf present) becomes true after init/join |
| `kubeadm init` failure | `vagrant ssh otel-control-plane -c "sudo journalctl -u kubelet -n 100"`; re-run with `sudo kubeadm init --config ... -v=5` | Pre-flight check failure (swap, ports, existing state from a prior partial init), or a version/image-pull issue | `sudo kubeadm reset -f` on the control plane, then re-run `vagrant provision otel-control-plane`; check the specific pre-flight error kubeadm prints | `is_control_plane_initialized` (`/etc/kubernetes/admin.conf` present) becomes true |
| Worker join token expired | `kubeadm join` output mentions an expired/invalid token | Tokens expire after ~24h by default; the worker's `.generated/cluster-info.env` is stale relative to the control plane's current token | Re-run `vagrant provision otel-control-plane` (regenerates `.generated/cluster-info.env` — `05-init-control-plane.sh` always refreshes it) then `vagrant provision otel-worker-1`/`-2` | `kubeadm join` succeeds; `is_node_joined` becomes true |
| Worker already joined | `07-join-worker.sh` logs `already joined ... skipping` | **Expected, not an error** — the script's idempotency guard | No action | Node still shows `Ready` in `kubectl get nodes` |
| Node `NotReady` | `kubectl describe node <name>`; `kubectl -n kube-system get pods -o wide \| grep cilium` | CNI (Cilium) not yet ready on that node, or containerd/kubelet issue on that node | Check the Cilium agent pod on that node first (`NotReady` almost always means "no CNI yet" on a freshly joined node); wait for the Cilium DaemonSet rollout | `kubectl get nodes` shows `Ready` |
| Wrong node IP | `kubectl get nodes -o wide` (check `INTERNAL-IP` column) | `/etc/default/kubelet`'s `--node-ip` wasn't applied before kubelet started, or was applied after | Confirm `01-configure-network.sh` ran before `04-install-kubernetes.sh`/kubelet start (fixed order in the Vagrantfile); `vagrant provision` and check `/etc/default/kubelet` on the node | `INTERNAL-IP` matches `config/cluster.env`'s value for that node |
| CoreDNS `Pending` | `kubectl -n kube-system get pods -l k8s-app=kube-dns` | No `Ready` node yet to schedule onto — usually resolves once Cilium is ready and at least one node is `Ready` | Wait for Cilium DaemonSet rollout and node readiness; re-check | CoreDNS pods reach `Running` |
| CoreDNS `CrashLoopBackOff` | `kubectl -n kube-system logs -l k8s-app=kube-dns --previous` | Often a DNS loop (CoreDNS forwarding to itself via the host's resolver) or a genuinely broken upstream resolver on the node | Check the node's `/etc/resolv.conf` for a loopback/`127.0.0.53`-style entry causing a forwarding loop; CoreDNS's `Corefile` may need a `forward` target adjustment (not currently customized by this module — investigate the actual log error first) | CoreDNS pods stable, no restart loop |
| Cilium agent failure | `kubectl -n kube-system logs -l k8s-app=cilium --previous`; `cilium status` | Kernel too old for a required eBPF feature, or `k8sServiceHost`/`k8sServicePort` misconfigured | Check the specific agent log error; confirm `config/cilium-values.yaml.tpl`'s rendered values in `.generated/rendered/cilium-values.yaml` | `kubectl -n kube-system rollout status daemonset/cilium` succeeds |
| Cilium operator failure | `kubectl -n kube-system logs -l name=cilium-operator --previous` | IPAM CIDR exhaustion (unlikely at this cluster's size) or apiserver connectivity issue | Check the specific operator log error | `kubectl -n kube-system rollout status deployment/cilium-operator` succeeds |
| Cilium connectivity test failure | `cilium connectivity test` (on the control plane, requires internet egress for the test's own images) | NetworkPolicy misconfiguration, or a genuine datapath issue | Narrow down with `hubble observe --verdict DROPPED` during the failing test | `cilium connectivity test` passes (note: this is a heavier, longer-running test than this module's own `tests/network-test.sh`) |
| Hubble Relay unavailable | `kubectl -n kube-system get pods -l k8s-app=hubble-relay` | Relay pod not scheduled/ready, or a port-forward wasn't actually established when `hubble status` was run | Confirm the Relay deployment is `Ready`; confirm the `kubectl port-forward` used by `make hubble-status` is actually running before querying | `hubble status` (via the port-forward) returns healthy |
| Pod DNS failure | `kubectl exec <pod> -- getent hosts kubernetes.default` | CoreDNS not ready yet, or a NetworkPolicy blocking DNS (port 53) egress from that pod | Confirm CoreDNS is `Running`; check for an overly broad deny-all `CiliumNetworkPolicy` in that pod's namespace | DNS resolution succeeds from the pod |
| Pod internet egress failure | `kubectl exec <pod> -- curl -fsS https://pkgs.k8s.io` | NAT/routing issue on the node, or a NetworkPolicy blocking egress | Confirm the node itself has internet egress (`vagrant ssh <node> -c curl ...`) before suspecting policy; check for a deny-all egress `CiliumNetworkPolicy` | `curl` from the pod succeeds |
| StorageClass missing | `kubectl get storageclass` | `08-install-storage.sh` never ran (e.g. `make setup` was interrupted before step 5/6) | Re-run: `vagrant ssh otel-control-plane -c "sudo NODE_NAME=otel-control-plane NODE_ROLE=control-plane bash /vagrant/scripts/guest/08-install-storage.sh"` | `kubectl get storageclass local-path` exists |
| PVC `Pending` | `kubectl describe pvc <name>`; `kubectl -n local-path-storage get pods` | local-path-provisioner pod not `Running`, or no schedulable node available yet | Confirm the provisioner Deployment is `Ready`; confirm at least one worker is `Ready` | PVC reaches `Bound` |
| Host cannot reach Kubernetes API | `kubectl --kubeconfig=.generated/kubeconfig get nodes` | Cluster isn't up, host-only routing issue, or a stale/corrupted exported kubeconfig | `make status` to check VM/cluster state first; `make kubeconfig` to re-validate/re-export | `kubectl get nodes` succeeds from the host |
| Kubeconfig permission failure | `ls -la .generated/kubeconfig` | Permissions were widened by some external tool/editor | `chmod 600 .generated/kubeconfig` (or re-run `make kubeconfig`, which sets this) | File mode is `600` |
| Kubernetes certificate issue | `vagrant ssh otel-control-plane -c "sudo kubeadm certs check-expiration"` | Cluster certs approaching/past their expiry (kubeadm defaults to 1 year for most, not an issue for a short-lived lab, but relevant if left running a long time) | `sudo kubeadm certs renew all` on the control plane, then restart control-plane static pods | `kubeadm certs check-expiration` shows no `EXPIRED`/`INVALID` entries |
| System clock drift | `vagrant ssh <node> -c "timedatectl status"` | Host suspended/resumed without the guest's clock catching up promptly | `vagrant ssh <node> -c "sudo systemctl restart chrony"`; wait a few seconds and re-check | `timedatectl status` shows `System clock synchronized: yes` |
| `vagrant provision` rerun failure | Re-run the specific failing guest script manually with `bash -x` for detail: `vagrant ssh <node> -c "sudo bash -x /vagrant/scripts/guest/<script>.sh"` | Usually a state assumption violated by a partial prior failure (e.g. apt lock held by an interrupted install) | Check `vagrant ssh <node> -c "sudo fuser /var/lib/dpkg/lock-frontend"`-style apt-lock issues first; scripts are designed to be idempotent, so a clean re-run should normally succeed once the underlying blocker is cleared | The specific script completes with its `[PASS] ... complete for <node>` line |
