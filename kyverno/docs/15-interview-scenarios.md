# Interview Scenarios

30 senior-level, scenario-based Kyverno questions. Each answer follows: Investigation → Root cause → Immediate mitigation → Permanent fix → Validation → Prevention. These are deliberately not definition-only questions ("what is a ClusterPolicy") — assume that baseline and focus on judgment under realistic, messy conditions.

---

**1. Every `kubectl apply` in the cluster suddenly starts failing with a webhook timeout error. What do you do, in order?**
Investigation: check Kyverno admission-controller pod status (`kubectl -n kyverno get pods`) and the webhook Service's endpoints first — is Kyverno actually down, or just slow? Root cause: commonly either a crashed/OOM-killed controller, or a genuinely overloaded one hitting `timeoutSeconds`. Mitigation: if it's a full outage and time-critical, and you have break-glass access, temporarily patch the webhook's `failurePolicy` to `Ignore` — document that you did this and why, since it removes protection cluster-wide until reverted. Permanent fix: restore controller health (scale up, fix the crash cause, right-size resources) and revert `failurePolicy` to `Fail`. Validation: `make validate-installation` (or your cluster's equivalent) passes, and a trivial `kubectl apply` succeeds normally. Prevention: HA (docs/11) sized *before* you need it, and alerting on webhook latency/availability, not just pod restarts.

**2. A team wants to skip Audit mode and go straight to Enforce for a new policy on a production namespace. How do you respond?**
Investigation: is there existing data (from a similar policy, or a dry-run via `kyverno apply` against a snapshot of real resources) suggesting the blast radius is actually small? Root cause of the request: usually time pressure, not bad intent. Mitigation: offer a fast-tracked but still real Audit window — even 24-48 hours of Audit data materially de-risks the Enforce flip and costs little. Permanent fix: standardize the audit-first rollout (root docs/DECISIONS.md ADR-013) as policy for the team, not a case-by-case negotiation. Validation: review the Audit-mode `PolicyReport` fail count and specific offending resources before flipping. Prevention: make the audit-first path the *easy* path (this lab's own paired audit/enforce policy files are exactly that pattern) so skipping it is more work, not less.

**3. A policy you assumed was narrow turns out to match far more resources than intended — production Deployments are now failing admission. What's your immediate move?**
Investigation: `kubectl describe clusterpolicy <name>` and the actual `match`/`exclude` block — usually a missing `kinds` restriction or an `exclude` that doesn't cover what you assumed. Root cause: policy was never tested against a representative resource set before enforce. Mitigation: immediately either revert to Audit mode or add a temporary, narrowly-scoped `PolicyException` for the specific blocked resources while you fix the `match`. Permanent fix: correct the `match`/`exclude`, add the missed cases as test resources in `tests/cli-test-cases/` so this can't silently regress. Validation: `kyverno test` against the corrected policy and the previously-blocked resources. Prevention: no policy goes to Enforce without CLI test coverage for both a compliant and non-compliant case, and an Audit-mode window first.

**4. A specific production workload is blocked by a policy that is, in principle, correct. The team says it's urgent. Walk through your decision process.**
Investigation: is the workload actually non-compliant for a good reason (genuine security gap) or a false positive (the policy's pattern doesn't account for a valid variant)? Root cause determines the whole response. Mitigation: if genuinely urgent and the workload is legitimately non-compliant but the risk is acceptable short-term, a narrowly-scoped, time-boxed `PolicyException` naming the exact resource — never a policy-wide loosening. Permanent fix: either fix the workload to comply, or fix the policy if it was a false positive; the exception should expire, not become permanent by default. Validation: `tests/exception-tests.sh`-style negative-case check confirming the exception didn't accidentally widen coverage. Prevention: docs/09's expiration/approval annotation convention, actually enforced by a scheduled check, not just documented.

**5. You inherit a cluster with 40 active PolicyExceptions, half with no expiration annotation set. How do you approach cleaning this up?**
Investigation: `kubectl get policyexceptions -A -o json` and cross-reference `expires`/`approved-by` annotations — how many are missing, how many are past-due. Root cause: exceptions were created without the governance convention (docs/09) actually being enforced. Mitigation: none needed immediately unless a specific exception is actively risky (overly broad scope) — audit first, don't mass-delete blind. Permanent fix: retroactively review each, either add proper annotations with real owner sign-off or revoke; going forward, add a CI/scheduled check that fails/pages on missing or expired annotations. Validation: `kubectl get policyexceptions -A` shows 100% annotation coverage after cleanup. Prevention: bake the annotation requirement into PR review/admission for `PolicyException` creation itself (e.g., a meta-policy requiring those annotations be set).

**6. Background scanning shows a resource as compliant, but you're certain it violates the policy when you look at it manually. What's going on?**
Investigation: check the policy's `background: true`/`false` setting, and the actual `resyncPeriod` — has enough time passed since the policy or the resource last changed? Root cause: usually either the resync hasn't caught up yet, or (for a `mutate` policy specifically) the compliant-looking state was achieved at admission time and background scanning doesn't apply to mutate rules retroactively at all (docs/03). Mitigation: none needed if it's just a resync-timing gap — wait it out. Permanent fix: if it's a genuine mismatch (not a timing issue), check for a stale/cached report entry from before the resource last changed. Validation: `kubectl describe policyreport` timestamp vs. resource's own `resourceVersion`/last-modified time. Prevention: understand and document (as this lab does) exactly which rule types background scanning does and doesn't retroactively affect.

**7. A generate policy's target resource keeps drifting from what the policy defines — someone keeps "fixing" it manually and it keeps reverting. How do you resolve the actual disagreement, not just the symptom?**
Investigation: is `synchronize: true` set (it reverts by design) and is the manual edit actually a legitimate, permanent requirement, or a one-off hotfix? Root cause: usually a mismatch between what the policy's `data` says and what's actually needed long-term. Mitigation: stop manually editing the generated resource — it will keep reverting for as long as `synchronize: true` is set. Permanent fix: if the manual edit should be permanent, update the *policy's* `data` block through normal review, not the generated resource directly. Validation: after the policy update, the generated resource matches without manual intervention and stays that way. Prevention: document generated-resource ownership clearly (docs/07) so "don't hand-edit this, it's generated" is obvious to anyone who encounters it.

**8. A cleanup policy is scoped more broadly than intended and has already deleted resources it shouldn't have. What now?**
Investigation: `kubectl -n <ns> get cleanuppolicy <name> -o yaml` — exactly what did `match`/`conditions` actually select? Check for any backup/recovery path for the deleted resources (this depends entirely on what was deleted — stateless workloads recreate trivially; anything with unique state may not). Root cause: selector was broader than the author verified before enabling the schedule. Mitigation: immediately delete/disable the CleanupPolicy to stop further deletions while you assess damage. Permanent fix: narrow the selector, prefer namespaced `CleanupPolicy` over cluster-wide, and require a dry-run-style manual review (`kubectl get <kind> -l <selector>`) of the actual match set before any cleanup policy goes live. Validation: re-verify the corrected selector's match set is exactly what's intended, nothing more. Prevention: root docs/DECISIONS.md ADR-016 exists precisely for this — narrow scoping is not optional guidance, it's the rule.

**9. Leadership wants a weekly policy-compliance report. How do you build it without adding new infrastructure?**
Investigation: what data do they actually want — pass/fail counts by team/namespace, trend over time, top violated policies? Root cause of the ask: usually visibility they currently don't have, not a specific tool preference. Mitigation/solution: `PolicyReport`/`ClusterPolicyReport` already has this data; a scheduled `kubectl get clusterpolicyreports -o json | jq ...` job (this lab's `scripts/collect-policy-reports.sh` is exactly this pattern) can produce the report with zero new components. Permanent fix: if the org later wants trend-over-time (not just point-in-time), that's when a real observability stack (out of scope for this independent lab, see docs/10's "Metrics" section) becomes justified — don't add it before the need is real. Validation: the report's numbers match `kubectl get policyreport -A` manually cross-checked. Prevention: n/a — this is a build question, not an incident.

**10. A `verifyImages` policy is rejecting a genuinely signed, legitimate image. How do you debug it?**
Investigation: `kyverno apply <policy> -r <resource>` for policy syntax first, then controller logs for the actual verification error (signature mismatch, wrong identity, registry error). Root cause: commonly either the `imageReferences` glob doesn't match the real image reference format, or (for keyless) the `subject`/`issuer` pattern doesn't match the actual signing identity. Mitigation: if urgent, a narrowly-scoped `PolicyException` for that exact image/resource while you investigate — never a blanket disable of image verification. Permanent fix: correct the `imageReferences`/attestor configuration once the real mismatch is identified. Validation: the same image is now admitted normally, and a genuinely-unsigned image is still rejected (regression-check both directions). Prevention: docs/08's two-path lab design — validate policy syntax offline before ever relying on it live.

**11. The image registry your `verifyImages` policy depends on goes down. What's the blast radius, and how do you handle it gracefully?**
Investigation: is this policy in `Audit` or `Enforce` mode, and does it match a narrow image set or broadly? Root cause: external dependency outage, outside your control. Mitigation: if Enforce and the outage is blocking legitimate, urgent deployments of otherwise-trusted images, a temporary, narrowly-scoped exception is the least-bad option — document it as tied to the specific outage, with a plan to revert once the registry recovers. Permanent fix: none — this is an inherent trade-off of live image verification; the mitigation is operational readiness, not a policy change. Validation: once the registry recovers, revert any temporary exceptions and confirm verification resumes normally. Prevention: scope `verifyImages` rules to the specific images that genuinely need it (as this lab does — only `ghcr.io/kyverno/*`) rather than every image cluster-wide, so a registry-specific outage has a correspondingly narrow blast radius.

**12. Admission latency has crept up 300ms over the last month with no obvious single cause. How do you find it?**
Investigation: per-policy/per-rule latency metrics if available (docs/10's metrics endpoints); if not instrumented yet, that's finding #1 — you can't diagnose what you can't measure. Root cause candidates in order of likelihood: a new or modified policy with a `context.apiCall`, growing policy count without corresponding controller scaling, or CPU throttling from under-sized resource requests as load grew. Mitigation: none urgent at 300ms unless it's approaching `timeoutSeconds`. Permanent fix: identify and optimize/remove the specific expensive rule(s), or scale admission-controller replicas/resources if it's genuinely aggregate load, not one bad rule. Validation: latency returns to baseline after the fix. Prevention: treat admission-controller latency as a first-class SLO with alerting, reviewed on every new policy merge (docs/13).

**13. You need to scale Kyverno for a cluster growing from 50 to 500 nodes. What actually needs to scale, and what doesn't?**
Investigation: which controller's load is actually tied to node/resource count vs. request rate vs. policy count? Root cause/reasoning: admission-controller load scales with *admission request rate*, not node count directly (though more nodes usually means more workloads means more requests); background-controller load scales with *matching resource count* × policy count ÷ resyncPeriod, which does scale with cluster size directly. Mitigation/fix: scale admission-controller replicas based on observed request rate and latency, not a fixed ratio to node count; consider whether every `background: true` policy still needs to be at the same resyncPeriod at 10x the resource count. Validation: load-test or gradually ramp and watch latency/resource metrics at each stage. Prevention: capacity-plan explicitly (docs/13) rather than assuming Kyverno scales linearly and transparently with cluster size without attention.

**14. Should a new enforce-mode policy use `failurePolicy: Fail` or `Ignore`? How do you decide?**
Investigation: is Kyverno itself genuinely HA in this cluster yet (docs/11), and how severe is the risk this specific policy protects against? Root cause of the question: `Fail` without real HA trades availability risk for security guarantee that isn't actually reliable yet. Mitigation/decision framework: for a cluster without proven Kyverno HA, start new enforce policies with `Ignore` and build toward `Fail` once HA (replicas, PDB, anti-affinity) is demonstrated under real failure conditions, not just configured. Permanent fix: n/a — this is an ongoing risk calculus per policy severity, not a one-time setting. Validation: chaos-test (kill an admission-controller replica) and confirm no admission outage before trusting `Fail` in production. Prevention: document this decision explicitly per policy (severity annotation, docs/11) rather than defaulting blindly to whatever the chart ships.

**15. A namespace exclusion someone added months ago is now covering three teams' production namespaces that were never meant to be excluded. How did this happen and how do you fix it?**
Investigation: `git blame`/PR history on the exclusion change — was it a specific name, or a pattern/label that new namespaces could match unintentionally? Root cause: almost always a wildcard or label-based exclusion (docs/12's "why broad exclusions weaken governance") rather than an explicit name list. Mitigation: none urgent unless something actively unsafe is running unpoliced in those namespaces — assess first. Permanent fix: convert to explicit, reviewed namespace names (this lab's own `config/namespaces.env` pattern) rather than a pattern/label any team could inadvertently or deliberately match. Validation: confirm the three teams' namespaces are now correctly policy-governed, and re-run background scans to catch anything that accumulated unpoliced. Prevention: treat exclusion-list changes with the same review rigor as the policies themselves — it's arguably higher-leverage than any single policy.

**16. Explain the overlap between Kyverno and Pod Security Admission to a skeptical platform lead who wants to "just use one."**
Answer: they solve overlapping but genuinely different problems (docs/12's full comparison) — PSA is free (no extra component, no network round-trip) but fixed to three profiles; Kyverno is a real dependency (docs/11's HA concerns) but arbitrarily expressive. The honest answer to "just use one": PSA alone can't express org-specific rules (required labels, image provenance, generation) at all — it's not a matter of preference, it's a capability gap. Kyverno alone, without PSA, means giving up a zero-dependency, always-on baseline for the cases PSA already covers well. Most mature platforms run both, layered, not as a redundant belt-and-suspenders but because they cover genuinely different ground.

**17. How does Kyverno compare to OPA/Gatekeeper, and how would you justify a choice between them to your team?**
Answer: functionally comparable as general admission policy engines; the real-world decision driver is usually team fluency (Rego, a genuinely separate language, vs. Kubernetes-native YAML/JMESPath) and specific feature needs (Kyverno's `generate`/`mutate`/`verifyImages` are more first-class; Gatekeeper's constraint-template separation is more explicit about policy-logic-vs-parameters reuse). Justify the choice on your team's actual composition and existing investment, not a "which is objectively better" framing — both are production-proven, CNCF-graduated (Kyverno) or CNCF-incubating-lineage tools.

**18. How do you deploy Kyverno policies via GitOps, and what breaks if you don't?**
Answer: policies are plain Kubernetes YAML — commit to git, sync via your GitOps controller of choice (ArgoCD/Flux/etc.), same as any other manifest. What breaks without GitOps: policy drift becomes invisible (someone `kubectl edit`s a live ClusterPolicy, and nothing catches it), rollback has no clean mechanism (you're reconstructing what changed from memory or audit logs instead of `git revert`), and review/approval has no natural gate. This lab's own policies (`kyverno/policies/`) are designed to be applied exactly this way — version-controlled, `kubectl apply -f`/GitOps-synced, never hand-edited live.

**19. Walk through your Kyverno upgrade strategy for a chart major-version bump.**
Investigation: read the specific chart's release notes for CRD schema changes and breaking changes first — never assume patch-level safety on a major bump. Root cause of upgrade risk: Helm does not always auto-upgrade CRDs on `helm upgrade` (a well-known Helm limitation, not Kyverno-specific) — an old CRD schema alongside a new controller version is a common self-inflicted incident. Mitigation/fix: follow the chart's documented CRD upgrade procedure explicitly (usually a separate `kubectl apply` of CRD manifests before the Helm upgrade). Validation: `make validate-installation`-equivalent full functional probe suite after the upgrade, not just "pods are Running." Prevention: test major upgrades in a non-production cluster first, and never skip the CRD-upgrade step because "it worked without it last time" (schema changes aren't present in every release, but you won't know without checking).

**20. Multiple teams want ownership over different policy sets in the same cluster. How do you structure this without chaos?**
Answer: namespaced `Policy` objects (not `ClusterPolicy`) scoped per-team namespace let a team own policies affecting only their own namespace without touching cluster-wide state; for genuinely cluster-wide concerns, a single centrally-owned `ClusterPolicy` set with clear CODEOWNERS-style review (docs/11's governance section) is more appropriate than letting every team touch cluster-scoped policy objects. The key structural decision: cluster-wide baseline (owned centrally, high review bar) vs. team-scoped policy (owned by the team, lighter review, blast-radius-limited to their own namespace by construction).

**21. How would you keep policy behavior consistent across multiple clusters?**
Answer: the same GitOps discipline (question 18) applied per-cluster from a shared source of policy YAML, with cluster-specific overlays only where genuinely necessary (e.g., different `LAB_PROFILE`-style resource sizing, not different security rules). The risk to explicitly guard against: policies drifting per-cluster because each was hand-tuned independently over time — a shared, version-controlled base with reviewed, minimal per-cluster deltas (this repository's own `values-minimum.yaml`/`values-recommended.yaml` pattern, generalized) keeps that from happening silently.

**22. Your Kyverno-governed cluster needs a disaster-recovery test. What do you actually need to prove works?**
Answer: everything in docs/11's Disaster Recovery table — policies restore from Git (`kubectl apply -f policies/`), the Helm release recreates identically (`make install`), exceptions restore the same way (they're just YAML), and generated resources self-heal once their source policy is restored (no manual restoration of generated resources needed or attempted). The actual test: wipe a non-production cluster's Kyverno installation entirely, run the exact recovery commands, then run the same functional validation (`make validate-installation`) used for original setup — if that passes identically, recovery is proven, not assumed.

**23. How do you test a new policy before it ever touches a live cluster?**
Answer: `kyverno test`/`kyverno apply` (docs/DECISIONS.md ADR-015, this lab's `tests/cli-test-cases/`) against both a compliant and non-compliant fixture resource, checked into version control alongside the policy itself — entirely offline, no cluster required. This catches syntax errors and pattern-logic mistakes before Audit mode even starts, which is strictly earlier and cheaper than catching them via a live `PolicyReport` review.

**24. What RBAC boundaries should exist around who can create/modify Kyverno policies, and why does it matter more than it might seem?**
Answer: `ClusterPolicy`/`Policy`/`PolicyException` creation is, functionally, a way to affect admission behavior cluster-wide (or namespace-wide) — equivalent in blast-radius terms to modifying RBAC itself, and should be gated with comparable care (limited to platform-team ServiceAccounts/users, reviewed via GitOps PR process, not broadly grantable). The easy-to-miss risk: a team with permission to create `PolicyException` objects in their own namespace, if scoped too loosely, could exempt themselves from cluster-wide policies without any centralized review — RBAC on `PolicyException` creation specifically deserves the same scrutiny as the policies it can exempt.

**25. Describe your incident-response process for "Kyverno is blocking legitimate traffic during an active production incident."**
Answer: first, confirm it's actually Kyverno and not a coincidental unrelated issue (check the specific denial message — Kyverno's messages are designed to be actionable, per this lab's own policy-writing convention of always including a clear `message`). If confirmed, the fastest safe mitigation is a narrowly-scoped, time-boxed `PolicyException` for the exact blocked resource (question 4) — not disabling the policy, not setting `failurePolicy: Ignore` globally, both of which remove protection far beyond the one blocked deployment. Document the exception with the incident reference in its annotations, and follow up post-incident on whether the policy or the workload needs a permanent fix.

**26. A `foreach` rule with a `preconditions` block is silently not catching a case you expected it to. How do you debug precondition logic specifically?**
Investigation: recall that a rule whose preconditions don't match is *skipped entirely* — no fail, no report entry, nothing (docs/04, docs/15/lab-15) — so "silently not catching" often means the precondition itself evaluated false, not that the rule ran and passed incorrectly. Root cause: usually a JMESPath expression referencing a field that doesn't exist on some resources (e.g., `request.object.spec.replicas` on a resource where `replicas` was never set, defaulting differently than expected). Mitigation/fix: add an explicit default in the JMESPath expression (`|| \`1\`` pattern, as this lab's own `policies/advanced/foreach-precondition-jmespath.yaml` uses) rather than assuming a field is always present. Validation: `kyverno test` with a fixture resource specifically missing that field. Prevention: always test the "field absent" case explicitly, not just the "field present with an unexpected value" case.

**27. How do you decide whether something belongs in a `pattern`/`anyPattern` block versus a `deny.conditions` block?**
Answer: `pattern` is the right tool when the check is naturally "does this resource's shape look like X" — structural, presence/absence, simple wildcard matching. `deny.conditions` is the right tool when the check is a genuine boolean expression — numeric comparisons, regex, list-intersection checks, anything that doesn't map cleanly onto "does this look like a template." This lab's own policies show the split concretely: label/resource-shape checks use `pattern` (`require-labels-enforce.yaml`); tag-regex and capability-list checks use `deny.conditions` (`restrict-latest-tag.yaml`, the capabilities rule in `restrict-privileged-containers.yaml`) — because neither of those is expressible as a clean structural pattern.

**28. Someone proposes disabling `background` scanning cluster-wide to reduce load. What do you tell them?**
Answer: that trade-off means giving up all visibility into pre-existing, non-compliant resources for every policy affected — new admission-time enforcement still works, but you lose the ability to ever know what's already out of compliance, cluster-wide, in one place. If the actual goal is reducing load, the more targeted fix is disabling `background` on the *specific* expensive policies (heavy `context.apiCall`, complex `foreach`) rather than everything — most policies' background scan cost is negligible; a blanket disable throws away real signal to fix a problem usually concentrated in a small number of rules (docs/13).

**29. A policy passes all your offline `kyverno test` cases but behaves differently once actually installed on the cluster. Why might that happen, and how do you close the gap?**
Answer: offline CLI testing validates policy syntax and dry-run pattern/deny logic against the fixture resources you provided — it does not fully reproduce live, network-dependent behavior: `context.apiCall` results depend on actual live cluster state the CLI can't fully simulate; `verifyImages` real signature verification requires reaching an actual registry/Rekor; background-scan timing is inherently a live-cluster phenomenon. Close the gap by treating CLI tests as necessary-but-not-sufficient for any rule using those features, and always following up with a real runtime test (this lab's `tests/*-policy-tests.sh`, run against an actual cluster) before considering such a policy fully validated (docs/08's explicit two-path design is this exact lesson, applied).

**30. You're asked to justify, to a security-skeptical stakeholder, why Audit mode has any value at all if it doesn't actually block anything.**
Answer: Audit mode's value isn't blocking — it's *measurement with zero deployment risk*. It turns "we think this policy is safe to enforce" from a guess into a data-backed decision: exact fail counts, exact offending resources, exact teams affected, collected against real production traffic without a single legitimate deployment ever being rejected. The alternative — going straight to Enforce — means your first real data point about a policy's blast radius is a production incident. For a security-skeptical stakeholder specifically, frame it as risk reduction on the *rollout*, not a weakening of the eventual control: every policy in this lab that ships in Enforce mode either started in, or still has, an Audit-mode twin (root docs/DECISIONS.md ADR-013) for exactly this reason.
