# Policy Rule Anatomy

Reference for every field this lab's policies (`../policies/`) actually use. Each entry: what it does, why it exists, and where to see it in this repo. Deep dives on `validate`/`mutate`/`generate`/`verifyImages` themselves live in their own numbered docs (05–08); this document covers the surrounding structure every rule shares.

## Autogen rules (a real Kyverno behavior worth knowing before it surprises you)

A `validate` rule matching only `kinds: ["Pod"]` is not actually limited to bare Pods — Kyverno automatically generates additional rules (visible with an `autogen-` name prefix, e.g. `autogen-validate-resources`) that apply the same pattern logic to the pod template inside Deployments, DaemonSets, StatefulSets, Jobs, and CronJobs. This is on by default and is *why* `policies/validate/require-resource-limits.yaml` — whose `match` only lists `kinds: ["Pod"]` — correctly rejects `demo/test-resources/noncompliant-deployment.yaml` (a Deployment, not a Pod) in `tests/cli-test-cases/resource-limits/`: the autogenerated `autogen-validate-resources` rule is what actually evaluated it, confirmed directly by running `kyverno test` (its output names the autogen rule explicitly). If you ever want a rule to apply *only* to bare Pods and never to controller-managed ones, autogen must be explicitly disabled per-rule (`skip-controllers`/autogen-controls annotation, chart/version-dependent syntax) — this lab's policies rely on the default (autogen on) throughout, which is almost always what you actually want.

## `match` / `exclude`

`match` decides which resources a rule even considers; `exclude` carves out exceptions from that match. Both support `any` (OR — any one block matching is enough) and `all` (AND — every block must match) at the top level, each containing `resources` blocks (`kinds`, `namespaces`, `names`, `selector`, `operations`). Every policy in this lab uses `match.any.resources.kinds` as its baseline filter — see `policies/validate/require-labels-enforce.yaml` for the simplest form.

**Common failure**: an `exclude` block that's broader than intended (e.g., excluding by `namespaces` when you meant to exclude by `names`) silently weakens the whole policy — always test the exclude path explicitly (`tests/cli-test-cases/`), not just the match path.

## `context`

Loads data a rule can reference via `{{ }}` variable substitution or JMESPath, from a source other than the admission request itself: `configMap` (a named ConfigMap in a given namespace — `policies/advanced/configmap-context-example.yaml`) or `apiCall` (an in-cluster Kubernetes API request, `urlPath` + optional `jmesPath` post-processing — `policies/advanced/api-call-context-example.yaml`). Context is resolved **before** `preconditions` and the rule's action run, in the order contexts are listed. This lab deliberately avoids any `apiCall` pointing outside the cluster (no internet dependency in policy decisions, per this module's own design constraint) — every `context` example here resolves against this cluster's own API server or its own ConfigMaps.

## `preconditions`

A gate evaluated after `context` resolves and before the rule's `validate`/`mutate`/`generate` action runs. A rule whose preconditions don't match is **skipped entirely** — no report entry is written, no denial happens, nothing. This is different from a `validate` rule that *runs* and fails. `policies/advanced/foreach-precondition-jmespath.yaml` has one rule per precondition dimension (`request.operation`, `request.namespace`, image registry, replica count, label presence) specifically to make this distinction concrete. See docs/15-jmespath-and-preconditions... (labs/lab-15) for the lab walkthrough.

## `foreach`

Iterates a rule's action over a JMESPath-selected list (most commonly `request.object.spec.containers[]`), applying `preconditions`/`pattern`/`deny.conditions`/`patchStrategicMerge` once per element, with `element` bound to the current item. Used in this lab wherever a check must apply per-container rather than per-Pod: `policies/validate/restrict-privileged-containers.yaml`'s capability check, `policies/mutate/add-security-context-defaults.yaml`'s container-level defaults, `policies/validate/restrict-latest-tag.yaml`'s per-container tag check.

## `deny`

An alternative to `pattern`/`anyPattern` for expressing "reject if condition(s) are true" using `conditions.any`/`conditions.all` with explicit operators (`Equals`, `NotEquals`, `In`, `NotIn`, `AnyIn`, `AllIn`, `AnyNotIn`, `GreaterThan`, etc.) rather than JMESPath-pattern matching. Better suited than `pattern` for conditions that aren't naturally expressed as "this field must look like X" — regex checks, numeric thresholds, list-intersection checks. Every `foreach` + capability/tag-check rule in this lab uses `deny.conditions` for exactly this reason.

## `pattern` / `anyPattern`

Kyverno's JMESPath-flavored structural pattern language: the resource must match `pattern` exactly in shape (with a small set of anchors — see below) for the rule to pass, or match *any one* of `anyPattern`'s alternatives. `?*` = "must be present and non-empty"; `*` = "must be present, any value including empty"; a literal glob string (`"registry.k8s.io/*"`) = wildcard string match.

**Anchors** (prefix on a field's key):
- `=(field)` — "conditional": only validated if the field is present in the resource; absent is fine.
- `+(field)` — mutate-only: "add if not present"; never overwrites an existing value (see docs/06-mutate-policies.md).
- `X(field)` — "existence negation": the field, if present, causes the rule to fail — used in `policies/validate/restrict-privileged-containers.yaml`'s hostPath rule.
- `^(field)` — "global": applies the check across the whole array rather than per-element (not used by this lab's policies, included here for completeness since you'll see it in the wild).

## `cel`

Kyverno's newer CEL-based policy types (`ValidatingPolicy`, `ImageValidatingPolicy`, etc. — GA since 1.17) express rule logic in Common Expression Language instead of JMESPath-pattern/`deny.conditions`. This lab does not use CEL as its primary teaching vehicle (root `docs/DECISIONS.md` ADR-018) since `ClusterPolicy`/`Policy` is still fully supported and more transferable to existing production Kyverno deployments and interview contexts today — but CEL is real, current, and the documented forward direction; see docs/02-architecture-and-internals.md's CRD table for where the CEL types fit.

## JMESPath, variables, request context

Every `{{ }}` block in a Kyverno policy is JMESPath evaluated against a merged context: `request` (the AdmissionReview — `request.object`, `request.namespace`, `request.operation`, `request.userInfo`), any named `context` entries, and (inside `foreach`) `element`. Kyverno extends standard JMESPath with helper functions used throughout this lab's policies: `split()`, `contains()`, `regex_match()`, `length()`. `docs/15-...`/`labs/lab-15-jmespath-and-preconditions.md` walks through evaluation order end to end.

## Namespace selectors and operation matching

`match.any.resources.namespaces` (a list, supports wildcards) restricts a rule to specific namespaces directly; `match.any.resources.selector` restricts by the *namespace's own labels* when combined with a namespace-kind match (as in `policies/generate/default-network-policy.yaml`, which matches `Namespace` resources by label, not by name). `match.any.resources.operations` (`CREATE`, `UPDATE`, `DELETE`, `CONNECT`) restricts which admission operations a rule even considers — `policies/advanced/api-call-context-example.yaml` uses `operations: ["CREATE"]` specifically so its (relatively expensive) API-call context only runs on new Deployments, not on every update to existing ones.
